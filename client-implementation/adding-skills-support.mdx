本指南详细介绍了如何为 AI 智能体或开发工具添加 Agent Skills 支持。它涵盖了完整的生命周期:发现技能、告知模型、将内容加载到上下文中,以及随着时间推移保持内容的有效性。

核心集成方式与你的智能体架构无关。实现细节取决于两个因素:

- **技能存放在哪里?** 本地运行的智能体可以扫描用户的文件系统以查找技能目录。云托管或沙箱化的智能体需要替代的发现机制——API、远程注册表或捆绑资产。
- **模型如何访问技能内容?** 如果模型具有文件读取能力,它可以直接读取 `SKILL.md` 文件。否则,你需要提供专用工具或以编程方式将技能内容注入到提示中。

本指南会注明这些差异的关键之处。你不需要支持每种场景——选择适合你的智能体的路径即可。

**前置条件**:熟悉 [Agent Skills 规范](/specification),该规范定义了 `SKILL.md` 文件格式、前置元数据字段和目录约定。

## 核心原则:渐进式披露

每个兼容技能的智能体都遵循相同的三层加载策略:

| 层级 | 加载内容 | 加载时机 | Token 成本 |
|------|----------|----------|-----------|
| 1. 目录 | 名称 + 描述 | 会话开始时 | 每个技能约 50-100 tokens |
| 2. 指令 | 完整的 `SKILL.md` 正文 | 技能被激活时 | &lt;5000 tokens(推荐) |
| 3. 资源 | 脚本、参考文档、资产 | 指令引用它们时 | 不定 |

模型从一开始就能看到目录,因此它知道有哪些技能可用。当它判断某个技能相关时,就会加载完整的指令。如果这些指令引用了支持文件,模型会根据需要单独加载它们。

这使得基础上下文保持较小,同时让模型能够按需访问专业知识。拥有 20 个已安装技能的智能体不会预先支付 20 个完整指令集的 token 成本——只有在特定对话中实际使用的技能才会产生成本。

## 步骤 1:发现技能

在会话启动时,查找所有可用的技能并加载它们的元数据。

### 扫描位置

要扫描哪些目录取决于你的智能体环境。大多数本地运行的智能体至少扫描两个范围:

- **项目级**(相对于工作目录):特定于项目或仓库的技能。
- **用户级**(相对于主目录):对给定用户的所有项目都可用的技能。

其他范围也是可能的——例如,由管理员部署的组织级技能,或与智能体本身捆绑的技能。正确的范围集取决于你的智能体的部署模型。

在每个范围内,考虑同时扫描**客户端特定目录**和 **`.agents/skills/` 约定**:

| 范围 | 路径 | 用途 |
|------|------|------|
| 项目 | `<project>/.<your-client>/skills/` | 你的客户端的原生位置 |
| 项目 | `<project>/.agents/skills/` | 跨客户端互操作性 |
| 用户 | `~/.<your-client>/skills/` | 你的客户端的原生位置 |
| 用户 | `~/.agents/skills/` | 跨客户端互操作性 |

`.agents/skills/` 路径已经成为跨客户端技能共享的广泛采用的约定。虽然 Agent Skills 规范没有规定技能目录必须位于何处(它只定义了目录内部的内容),但扫描 `.agents/skills/` 意味着其他兼容客户端安装的技能会自动对你的客户端可见,反之亦然。

<Note>
一些实现还会扫描 `.claude/skills/`(项目级和用户级)以实现实用的兼容性,因为许多现有技能都安装在那里。其他附加位置包括向上到 git 根目录的祖先目录(对 monorepo 有用)、[XDG](https://specifications.freedesktop.org/basedir-spec/latest/) 配置目录以及用户配置的路径。
</Note>

### 扫描内容

在每个技能目录中,查找**包含名为 `SKILL.md` 文件的子目录**:

```
~/.agents/skills/
├── pdf-processing/
│   ├── SKILL.md          ← 被发现
│   └── scripts/
│       └── extract.py
├── data-analysis/
│   └── SKILL.md          ← 被发现
└── README.md             ← 被忽略(不是技能目录)
```

实用的扫描规则:

- 跳过不会包含技能的目录,如 `.git/` 和 `node_modules/`
- 可选地遵守 `.gitignore` 以避免扫描构建产物
- 设置合理的边界(例如,最大深度 4-6 层,最多 2000 个目录)以防止在大型目录树中无限扫描

### 处理名称冲突

当两个技能共享相同的 `name` 时,应用确定性的优先规则。

现有实现中的通用约定:**项目级技能覆盖用户级技能。**

在同一范围内(例如,在 `<project>/.agents/skills/` 和 `<project>/.<your-client>/skills/` 下都发现了名为 `code-review` 的技能),先发现或后发现都是可以接受的——选择一个并保持一致。当发生冲突时记录警告,以便用户知道某个技能被覆盖了。

### 信任考虑

项目级技能来自正在处理的仓库,可能是不受信任的(例如,刚克隆的开源项目)。考虑在信任检查上设置项目级技能加载的门控——只有当用户将项目文件夹标记为受信任时才加载它们。这可以防止不受信任的仓库悄悄地向智能体的上下文中注入指令。

### 云托管和沙箱化智能体

如果你的智能体在容器或远程服务器中运行,它将无法访问用户的本地文件系统。发现需要根据技能范围以不同方式工作:

- **项目级技能**通常是最简单的情况。如果智能体在克隆的仓库上操作(即使在沙箱内),项目级技能会随代码一起传输,可以从仓库的目录树中扫描。
- **用户级和组织级技能**在沙箱中不存在。你需要从外部源提供它们——例如,克隆配置仓库,通过智能体的设置接受技能 URL 或包,或让用户通过 Web UI 上传技能目录。
- **内置技能**可以打包为智能体部署工件中的静态资产,使它们在每个会话中都可用,无需外部获取。

一旦智能体可以使用技能,生命周期的其余部分——解析、披露、激活——都以相同的方式工作。

## 步骤 2:解析 `SKILL.md` 文件

对于每个发现的 `SKILL.md`,提取元数据和正文内容。

### 前置元数据提取

`SKILL.md` 文件有两部分:在 `---` 分隔符之间的 YAML 前置元数据,以及闭合分隔符之后的 markdown 正文。解析方法:

1. 在文件开头找到开始的 `---` 和之后的结束 `---`。
2. 解析它们之间的 YAML 块。提取 `name` 和 `description`(必需),以及任何可选字段。
3. 闭合 `---` 之后的所有内容(修剪后)是技能的正文内容。

有关前置元数据字段及其约束的完整集合,请参阅[规范](/specification)。

### 处理格式错误的 YAML

为其他客户端编写的技能文件可能包含技术上无效的 YAML,但它们的解析器恰好接受。最常见的问题是包含冒号的未引用值:

```yaml
# 技术上无效的 YAML——冒号会破坏解析
description: Use this skill when: the user asks about PDFs
```

考虑一个回退方案,在重试之前将这些值包装在引号中或将它们转换为 YAML 块标量。这以最小的成本提高了跨客户端兼容性。

### 宽松验证

对问题发出警告,但仍尽可能加载技能:

- 名称与父目录名称不匹配 → 警告,仍然加载
- 名称超过 64 个字符 → 警告,仍然加载
- 描述缺失或为空 → 跳过技能(描述对于披露是必不可少的),记录错误
- YAML 完全无法解析 → 跳过技能,记录错误

记录诊断信息,以便可以向用户显示(在调试命令、日志文件或 UI 中),但不要因为表面问题而阻止技能加载。

<Note>
[规范](/specification)对 `name` 字段定义了严格的约束(匹配父目录、字符集、最大长度)。上述宽松方法故意放宽了这些约束,以提高与为其他客户端编写的技能的兼容性。
</Note>

### 存储内容

至少,每个技能记录需要三个字段:

| 字段 | 描述 |
|------|------|
| `name` | 来自前置元数据 |
| `description` | 来自前置元数据 |
| `location` | `SKILL.md` 文件的绝对路径 |

将这些存储在按 `name` 键控的内存映射中,以便在激活期间快速查找。

你还可以在发现时存储**正文**(前置元数据之后的 markdown 内容),或在激活时从 `location` 读取它。存储它可以使激活更快;在激活时读取它可以减少总体内存使用,并在激活之间获取技能文件的更改。

技能的**基础目录**(即 `location` 的父目录)稍后需要用于解析相对路径和枚举捆绑资源——在需要时从 `location` 派生它。

## 步骤 3:向模型披露可用技能

告诉模型存在哪些技能,而不加载它们的完整内容。这是[渐进式披露的第 1 层](#核心原则渐进式披露)。

### 构建技能目录

对于每个发现的技能,在适合你的技术栈的任何结构化格式中包含 `name`、`description` 和可选的 `location`(`SKILL.md` 文件的路径)——XML、JSON 或项目符号列表都可以:

```xml
<available_skills>
  <skill>
    <name>pdf-processing</name>
    <description>提取 PDF 文本、填充表单、合并文件。在处理 PDF 时使用。</description>
    <location>/home/user/.agents/skills/pdf-processing/SKILL.md</location>
  </skill>
  <skill>
    <name>data-analysis</name>
    <description>分析数据集、生成图表并创建摘要报告。</description>
    <location>/home/user/project/.agents/skills/data-analysis/SKILL.md</location>
  </skill>
</available_skills>
```

`location` 字段有两个用途:它启用文件读取激活(参见[步骤 4](#步骤-4激活技能)),并为模型提供解析技能正文中相对引用(如 `scripts/evaluate.py`)的基础路径。如果你的专用激活工具在其结果中提供了技能目录路径(参见步骤 4 中的[结构化包装](#结构化包装)),你可以从目录中省略 `location`。否则,请包含它。

每个技能向目录添加大约 50-100 个 tokens。即使安装了数十个技能,目录也保持紧凑。

### 目录放置位置

两种方法很常见:

**系统提示部分**:将目录作为系统提示中的标记部分添加,前面有关于如何使用技能的简要说明。这是最简单的方法,适用于任何具有文件读取工具访问权限的模型。

**工具描述**:将目录嵌入专用技能激活工具的描述中(参见[步骤 4](#步骤-4激活技能))。这使系统提示保持干净,并自然地将发现与激活耦合。

两者都可以。系统提示放置更简单且更广泛兼容;当你有专用激活工具时,工具描述嵌入更干净。

### 行为指令

在目录旁边包含一个简短的指令块,告诉模型如何以及何时使用技能。措辞取决于你支持的激活机制(参见[步骤 4](#步骤-4激活技能)):

**如果模型通过读取文件来激活技能:**

```
以下技能为特定任务提供专业指令。
当任务与技能的描述匹配时,使用你的文件读取工具在继续之前加载
列出位置的 SKILL.md。
当技能引用相对路径时,针对技能的目录(SKILL.md 的父目录)
解析它们,并在工具调用中使用绝对路径。
```

**如果模型通过专用工具激活技能:**

```
以下技能为特定任务提供专业指令。
当任务与技能的描述匹配时,使用技能的名称调用 activate_skill 工具
以加载其完整指令。
```

保持这些指令简洁。目标是告诉模型技能的存在以及如何加载它们——技能内容本身在加载后提供详细的指令。

### 过滤

某些技能应从目录中排除。常见原因:

- 用户已在设置中禁用该技能
- 权限系统拒绝访问该技能
- 技能已选择退出模型驱动的激活(例如,通过 `disable-model-invocation` 标志)

**完全隐藏过滤的技能**,而不是列出它们并在激活时阻止。这可以防止模型浪费时间尝试加载它无法使用的技能。

### 当没有可用技能时

如果没有发现技能,请完全省略目录和行为指令。不要显示空的 `<available_skills/>` 块或注册没有有效选项的技能工具——这会使模型感到困惑。

## 步骤 4:激活技能

当模型或用户选择技能时,将完整的指令传递到对话上下文中。这是[渐进式披露的第 2 层](#核心原则渐进式披露)。

### 模型驱动的激活

大多数实现依赖于模型自己的判断作为激活机制,而不是实现工具端的触发匹配或关键字检测。模型读取目录(来自[步骤 3](#步骤-3向模型披露可用技能)),判断技能与当前任务相关,然后加载它。

两种实现模式:

**文件读取激活**:模型使用目录中的 `SKILL.md` 路径调用其标准文件读取工具。不需要特殊基础设施——智能体现有的文件读取能力就足够了。模型将文件内容作为工具结果接收。当模型具有文件访问权限时,这是最简单的方法。

**专用工具激活**:注册一个工具(例如 `activate_skill`),它接受技能名称并返回内容。当模型无法直接读取文件时,这是必需的,即使可以,这也是可选的(但有用)。相对于原始文件读取的优势:

- 控制返回的内容——例如,剥离 YAML 前置元数据或保留它(参见下面的[模型接收的内容](#模型接收的内容))
- 将内容包装在结构化标签中以便在上下文管理期间识别
- 在指令旁边列出捆绑的资源(例如 `references/*`)
- 执行权限或提示用户同意
- 跟踪激活以进行分析

<Note>
如果你使用专用激活工具,请将 `name` 参数约束为有效技能名称的集合(例如,作为工具模式中的枚举)。这可以防止模型虚构不存在的技能名称。如果没有可用技能,则根本不要注册该工具。
</Note>

### 用户显式激活

用户还应该能够直接激活技能,而无需等待模型决定。最常见的模式是**斜杠命令或提及语法**(`/skill-name` 或 `$skill-name`),工具会拦截它。具体语法由你决定——关键思想是工具处理查找和注入,因此模型接收技能内容而无需自己采取激活操作。

自动完成小部件(在用户输入时列出可用技能)也可以使其可发现。

### 模型接收的内容

当技能被激活时,模型接收技能的指令。关于该内容确切外观的两个选项:

**完整文件**:模型看到整个 `SKILL.md`,包括 YAML 前置元数据。这是文件读取激活的自然结果,其中模型读取原始文件。这对于专用工具也是一个有效的选择。前置元数据可能包含在激活时有用的字段——例如,[`compatibility`](/specification#compatibility-field) 注明了可能影响模型如何执行技能指令的环境要求。

**仅正文(剥离前置元数据)**:工具解析并删除 YAML 前置元数据,仅返回 markdown 指令。在具有专用激活工具的现有实现中,大多数采用这种方法——在发现期间提取 `name` 和 `description` 后剥离前置元数据。

两种方法在实践中都有效。

### 结构化包装

如果你使用专用激活工具,请考虑将技能内容包装在识别标签中。例如:

```xml
<skill_content name="pdf-processing">
# PDF 处理

## 何时使用此技能
当用户需要处理 PDF 文件时使用此技能...

[SKILL.md 正文的其余部分]

技能目录: /home/user/.agents/skills/pdf-processing
此技能中的相对路径相对于技能目录。

<skill_resources>
  <file>scripts/extract.py</file>
  <file>scripts/merge.py</file>
  <file>references/pdf-spec-summary.md</file>
</skill_resources>
</skill_content>
```

这具有实际好处:

- 模型可以清楚地将技能指令与其他对话内容区分开来
- 工具可以在上下文压缩期间识别技能内容([步骤 5](#步骤-5随时间管理技能上下文))
- 捆绑的资源会向模型显示,而不会被急切地加载

### 列出捆绑资源

当专用激活工具返回技能内容时,它还可以枚举技能目录中的支持文件(脚本、参考文档、资产)——但**不应急切地读取它们**。当技能的指令引用它们时,模型使用其文件读取工具按需加载特定文件。

对于大型技能目录,请考虑限制列表并注明它可能不完整。

### 权限白名单

如果你的智能体有一个控制文件访问的权限系统,请**将技能目录加入白名单**,以便模型可以读取捆绑的资源而不会触发用户确认提示。如果没有这个,对捆绑脚本或参考文件的每次引用都会导致权限对话框,从而破坏包含 `SKILL.md` 本身之外资源的技能的流程。

## 步骤 5:随时间管理技能上下文

一旦技能指令进入对话上下文,在会话期间保持它们的有效性。

### 保护技能内容免受上下文压缩

如果你的智能体在上下文窗口填满时截断或总结旧消息,请**豁免技能内容不被修剪**。技能指令是持久的行为指导——在对话中间丢失它们会在没有任何可见错误的情况下悄悄地降低智能体的性能。模型继续运行,但没有技能提供的专业指令。

常见方法:

- 将技能工具输出标记为受保护,以便修剪算法跳过它们
- 使用步骤 4 中的[结构化标签](#结构化包装)来识别技能内容并在压缩期间保留它

### 去重激活

考虑跟踪当前会话中已激活的技能。如果模型(或用户)尝试加载已在上下文中的技能,你可以跳过重新注入,以避免相同的指令在对话中多次出现。

### 子智能体委派(可选)

这是一种高级模式,仅由某些客户端支持。不是将技能指令注入主对话,而是在**单独的子智能体会话**中运行技能。子智能体接收技能指令,执行任务,并将其工作摘要返回到主对话。

当技能的工作流程足够复杂以从专用的、集中的会话中受益时,此模式很有用。

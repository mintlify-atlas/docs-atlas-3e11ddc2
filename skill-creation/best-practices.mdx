## 从真实专业知识出发

技能创建中的一个常见陷阱是在不提供特定领域上下文的情况下让 LLM 生成技能——仅依赖 LLM 的通用训练知识。结果是模糊、泛泛的程序（"适当处理错误"、"遵循身份验证最佳实践"），而不是使技能有价值的特定 API 模式、边缘情况和项目约定。

有效的技能建立在真实专业知识之上。关键是将特定领域的上下文输入到创建过程中。

### 从实际任务中提取

在与智能体的对话中完成一个真实任务，在此过程中提供上下文、更正和偏好。然后将可重用的模式提取为技能。注意以下几点：

- **有效的步骤** — 导致成功的操作序列
- **你做出的更正** — 你引导智能体方法的地方（例如，"使用库 X 而不是 Y"、"检查边缘情况 Z"）
- **输入/输出格式** — 输入和输出的数据样子
- **你提供的上下文** — 智能体尚不知道的项目特定事实、约定或约束

### 从现有项目工件中综合

当你有大量现有知识时，可以将其输入到 LLM 中并要求它综合一个技能。从团队实际事件报告和运行手册综合的数据管道技能将优于从通用"数据工程最佳实践"文章综合的技能，因为它捕获了*你的*模式、失败模式和恢复程序。关键是项目特定的材料，而不是通用参考。

良好的源材料包括：

- 内部文档、运行手册和风格指南
- API 规范、模式和配置文件
- 代码审查评论和问题跟踪器（捕获重复出现的关注点和审查者期望）
- 版本控制历史记录，尤其是补丁和修复（通过实际更改的内容揭示模式）
- 真实世界的失败案例及其解决方案

## 通过实际执行进行优化

技能的第一稿通常需要优化。针对实际任务运行技能，然后将结果——所有结果，不仅仅是失败——反馈到创建过程中。问：什么触发了误报？遗漏了什么？可以删减什么？

即使一次执行后修订也能明显提高质量，复杂的领域通常需要几次迭代。

<Tip>
阅读智能体执行跟踪，而不仅仅是最终输出。如果智能体在无成效的步骤上浪费时间，常见原因包括：指令过于模糊（智能体在找到有效方法之前尝试多种方法）、指令不适用于当前任务（智能体仍然遵循）、或提供了太多选项而没有明确的默认值。
</Tip>

有关更结构化的迭代方法，包括测试用例、断言和评分，请参阅[评估技能输出质量](/skill-creation/evaluating-skills)。

## 明智地使用上下文

一旦技能激活，其完整的 `SKILL.md` 主体就会与对话历史、系统上下文和其他活动技能一起加载到智能体的上下文窗口中。技能中的每个标记都与该窗口中的其他所有内容竞争智能体的注意力。

### 添加智能体缺少的，省略它知道的

专注于智能体*没有*你的技能就不会知道的内容：项目特定的约定、领域特定的程序、不明显的边缘情况，以及要使用的特定工具或 API。你不需要解释什么是 PDF、HTTP 如何工作或数据库迁移做什么。

````markdown
<!-- 过于冗长 — 智能体已经知道什么是 PDF -->
## 提取 PDF 文本

PDF（便携式文档格式）文件是一种常见的文件格式，包含
文本、图像和其他内容。要从 PDF 中提取文本，你需要
使用一个库。推荐使用 pdfplumber，因为它可以很好地处理大多数情况。

<!-- 更好 — 直接跳到智能体自己不知道的内容 -->
## 提取 PDF 文本

使用 pdfplumber 进行文本提取。对于扫描文档，回退到
pdf2image 和 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

对每条内容问自己："没有这个指令，智能体会做错吗？"如果答案是否定的，就删除它。如果不确定，就测试它。如果智能体在没有技能的情况下已经很好地处理了整个任务，那么技能可能没有增加价值。请参阅[评估技能输出质量](/skill-creation/evaluating-skills)以了解如何系统地测试这一点。

### 设计连贯的单元

决定技能应该涵盖什么就像决定函数应该做什么：你希望它封装一个与其他技能良好组合的连贯工作单元。范围过窄的技能迫使为单个任务加载多个技能，存在开销和指令冲突的风险。范围过广的技能变得难以精确激活。查询数据库并格式化结果的技能可能是一个连贯的单元，而还涵盖数据库管理的技能可能试图做太多事情。

### 追求适度的细节

过于全面的技能可能弊大于利——智能体难以提取相关内容，可能会追求由不适用于当前任务的指令触发的无成效路径。简洁、逐步的指导和一个工作示例往往优于详尽的文档。当你发现自己涵盖了每个边缘情况时，考虑大多数是否更好地由智能体自己的判断来处理。

### 通过渐进式披露构建大型技能

[规范](/specification#progressive-disclosure)建议将 `SKILL.md` 保持在 500 行和 5,000 个标记以下——只是智能体在每次运行时需要的核心指令。当技能确实需要更多内容时，将详细的参考材料移动到 `references/` 或类似目录中的单独文件中。

关键是告诉智能体*何时*加载每个文件。"如果 API 返回非 200 状态码，请阅读 `references/api-errors.md`" 比通用的 "详情请参阅 references/" 更有用。这让智能体可以按需加载上下文而不是预先加载，这就是[渐进式披露](/what-are-skills#how-skills-work)设计的工作方式。

## 校准控制

技能的每个部分都不需要相同的规定性水平。使指令的具体性与任务的脆弱性相匹配。

### 将具体性与脆弱性相匹配

**给智能体自由**，当多种方法有效且任务容忍变化时。对于灵活的指令，解释*为什么*可能比严格的指令更有效——理解指令背后目的的智能体做出更好的上下文相关决策。代码审查技能可以描述要查找的内容，而无需规定确切的步骤：

```markdown
## 代码审查流程

1. 检查所有数据库查询是否存在 SQL 注入（使用参数化查询）
2. 验证每个端点的身份验证检查
3. 在并发代码路径中查找竞态条件
4. 确认错误消息不会泄露内部详细信息
```

**要有规定性**，当操作脆弱、一致性重要或必须遵循特定序列时：

````markdown
## 数据库迁移

严格按照以下顺序运行：

```bash
python scripts/migrate.py --verify --backup
```

不要修改命令或添加其他标志。
````

大多数技能都是混合的。独立校准每个部分。

### 提供默认值，而不是菜单

当多个工具或方法可以工作时，选择一个默认值并简要提及替代方案，而不是将它们作为同等选项呈现。

````markdown
<!-- 选项太多 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...

<!-- 清晰的默认值和退出通道 -->
使用 pdfplumber 进行文本提取：

```python
import pdfplumber
```

对于需要 OCR 的扫描 PDF，请改用 pdf2image 和 pytesseract。
````

### 偏向程序而非声明

技能应该教智能体*如何处理*一类问题，而不是*为特定实例产生什么*。比较：

```markdown
<!-- 具体答案 — 仅对这个确切任务有用 -->
将 `orders` 表与 `customers` 表在 `customer_id` 上连接，过滤
`region = 'EMEA'` 的地方，并对 `amount` 列求和。

<!-- 可重用的方法 — 适用于任何分析查询 -->
1. 从 `references/schema.yaml` 读取模式以找到相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何过滤器应用为 WHERE 子句
4. 根据需要聚合数字列并格式化为 markdown 表
```

这并不意味着技能不能包含特定的细节——输出格式模板（参见[输出格式模板](#templates-for-output-format)）、像"永远不要输出 PII"这样的约束，以及工具特定的指令都是有价值的。重点是*方法*应该泛化，即使个别细节是特定的。

## 有效指令的模式

这些是构建技能内容的可重用技术。不是每个技能都需要所有技术——使用适合你任务的技术。

### 输出格式模板

当你需要智能体以特定格式产生输出时，提供一个模板。这比用散文描述格式更可靠，因为智能体可以很好地与具体结构进行模式匹配。简短的模板可以在 `SKILL.md` 中内联；对于较长的模板，或仅在某些情况下需要的模板，将它们存储在 `assets/` 中并从 `SKILL.md` 中引用，以便它们仅在需要时加载。

````markdown
## 报告结构

使用此模板，根据需要为特定分析调整部分：

```markdown
# [分析标题]

## 执行摘要
[关键发现的一段概述]

## 关键发现
- 发现 1 及支持数据
- 发现 2 及支持数据

## 建议
1. 具体可操作的建议
2. 具体可操作的建议
```
````

### 多步骤工作流的检查清单

明确的检查清单帮助智能体跟踪进度并避免跳过步骤，尤其是当步骤具有依赖关系或验证门时。

```markdown
## 表单处理工作流

进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填充表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）
```

### 验证循环

指示智能体在继续之前验证自己的工作。模式是：做工作，运行验证器（脚本、参考检查清单或自检），修复任何问题，并重复直到验证通过。

```markdown
## 编辑工作流

1. 进行你的编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误消息
   - 修复问题
   - 再次运行验证
4. 只有在验证通过时才继续
```

参考文档也可以充当"验证器"——指示智能体在最终确定之前对照参考检查其工作。

### 计划-验证-执行

对于批处理或破坏性操作，让智能体以结构化格式创建中间计划，根据真实来源验证它，然后才执行。

```markdown
## PDF 表单填充

1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段名称、类型以及是否为必填）
2. 创建 `field_values.json`，将每个字段名称映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名称是否存在于表单中，类型是否兼容，以及
   必填字段是否缺失）
4. 如果验证失败，修订 `field_values.json` 并重新验证
5. 填充表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

关键成分是步骤 3：一个验证脚本，根据真实来源（`form_fields.json`）检查计划（`field_values.json`）。像"字段 'signature_date' 未找到 — 可用字段：customer_name、order_total、signature_date_signed"这样的错误为智能体提供了足够的信息来自我纠正。

### 捆绑可重用脚本

在[迭代技能](/skill-creation/evaluating-skills)时，比较跨测试用例的智能体执行跟踪。如果你注意到智能体在每次运行时独立重新发明相同的逻辑——构建图表、解析特定格式、验证输出——这是一个信号，表明要编写一个经过测试的脚本并将其捆绑在 `scripts/` 中。

有关设计和捆绑脚本的更多信息，请参阅[在技能中使用脚本](/skill-creation/using-scripts)。

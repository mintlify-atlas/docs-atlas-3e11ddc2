技能只有在被激活时才有帮助。`SKILL.md` 前置元数据中的 `description` 字段是智能体用来决定是否为给定任务加载技能的主要机制。描述不足意味着技能在应该触发时不会触发；描述过于宽泛意味着它在不应该触发时触发。

本指南介绍如何系统地测试和改进技能描述的触发准确性。

## 技能触发的工作原理

智能体使用[渐进式披露](/what-are-skills#how-skills-work)来管理上下文。在启动时，它们只加载每个可用技能的 `name` 和 `description`——刚好足以决定何时技能可能相关。当用户的任务与描述匹配时，智能体将完整的 `SKILL.md` 读入上下文并遵循其指令。

这意味着描述承担了触发的全部负担。如果描述没有传达技能何时有用，智能体就不会知道使用它。

一个重要的细微差别：智能体通常只在需要超出其单独处理能力的知识或能力的任务时才咨询技能。像"读取这个 PDF"这样简单的单步请求可能不会触发 PDF 技能，即使描述完全匹配，因为智能体可以使用基本工具处理它。涉及专业知识的任务——不熟悉的 API、领域特定的工作流或不常见的格式——是精心编写的描述可以产生影响的地方。

## 编写有效的描述

在测试之前，了解好的描述是什么样子会有所帮助。几个原则：

- **使用祈使措辞。** 将描述构建为对智能体的指令："在...时使用此技能"而不是"此技能做..."。智能体正在决定是否采取行动，所以告诉它何时行动。
- **关注用户意图，而不是实现。** 描述用户试图实现什么，而不是技能的内部机制。智能体与用户要求的内容进行匹配。
- **宁可咄咄逼人。** 明确列出技能适用的上下文，包括用户没有直接命名领域的情况："即使他们没有明确提到 'CSV' 或 '分析'"。
- **保持简洁。** 几句话到一个简短的段落通常是正确的——足够长以涵盖技能的范围，足够短以至于它不会在许多技能中膨胀智能体的上下文。[规范](/specification#description-field)强制执行 1024 个字符的硬限制。

## 设计触发评估查询

要测试触发，你需要一组评估查询——带有标签的真实用户提示，标明它们是否应该触发你的技能。

```json eval_queries.json
[
  { "query": "I've got a spreadsheet in ~/data/q4_results.xlsx with revenue in col C and expenses in col D — can you add a profit margin column and highlight anything under 10%?", "should_trigger": true },
  { "query": "whats the quickest way to convert this json file to yaml", "should_trigger": false }
]
```

目标是大约 20 个查询：8-10 个应该触发，8-10 个不应该触发。

### 应该触发的查询

这些测试描述是否捕获了技能的范围。沿几个轴变化它们：

- **措辞**：有些正式，有些随意，有些有拼写错误或缩写。
- **明确性**：有些直接命名技能的领域（"分析这个 CSV"），其他描述需求而不命名它（"我的老板想要从这个数据文件中得到一个图表"）。
- **细节**：混合简洁的提示和上下文丰富的提示——一个简短的"分析我的销售 CSV 并制作图表"与一个更长的消息，其中包含文件路径、列名和背景故事。
- **复杂性**：变化步骤数和决策点。包括单步任务和多步工作流，以测试智能体是否能够在任务被埋在更大的链中时辨别技能是否相关。

最有用的应该触发查询是那些技能会有帮助但从查询本身看不出联系的查询。这些是描述措辞产生影响的情况——如果查询已经要求技能所做的事情，任何合理的描述都会触发。

### 不应该触发的查询

最有价值的负面测试用例是**接近失误**——与你的技能共享关键词或概念但实际上需要不同内容的查询。这些测试描述是否精确，而不仅仅是宽泛。

对于 CSV 分析技能，弱的负面示例是：

- `"Write a fibonacci function"` — 明显不相关，不测试任何东西。
- `"What's the weather today?"` — 没有关键词重叠，太容易。

强的负面示例：

- `"I need to update the formulas in my Excel budget spreadsheet"` — 共享"电子表格"和"数据"概念，但需要 Excel 编辑，而不是 CSV 分析。
- `"can you write a python script that reads a csv and uploads each row to our postgres database"` — 涉及 CSV，但任务是数据库 ETL，而不是分析。

### 现实性提示

真实的用户提示包含通用测试查询缺少的上下文。包括：

- 文件路径（`~/Downloads/report_final_v2.xlsx`）
- 个人上下文（`"我的经理要求我..."`）
- 具体细节（列名、公司名称、数据值）
- 随意的语言、缩写和偶尔的拼写错误

## 测试描述是否触发

基本方法：通过安装了技能的智能体运行每个查询并观察智能体是否调用它。确保技能已注册并且智能体可以发现——这在不同客户端中的工作方式不同（例如，技能目录、配置文件或 CLI 标志）。

大多数智能体客户端提供某种形式的可观察性——执行日志、工具调用历史或详细输出——让你可以看到在运行期间咨询了哪些技能。查看客户端的文档以获取详细信息。如果智能体加载了你的技能的 `SKILL.md`，则技能触发；如果智能体在没有咨询它的情况下继续，则技能没有触发。

如果满足以下条件，查询"通过"：
- `should_trigger` 为 `true` 且技能被调用，或
- `should_trigger` 为 `false` 且技能未被调用。

### 多次运行

模型行为是非确定性的——同一查询可能在一次运行时触发技能，但在下一次不触发。多次运行每个查询（3 是一个合理的起点）并计算**触发率**：技能被调用的运行比例。

如果应该触发的查询的触发率高于阈值（0.5 是一个合理的默认值），则该查询通过。如果不应该触发的查询的触发率低于该阈值，则该查询通过。

对于 20 个查询，每个查询运行 3 次，总共是 60 次调用。你需要编写脚本来做这件事。这是一般结构——用你的智能体客户端提供的任何内容替换 `check_triggered` 中的 `claude` 调用和检测逻辑：

```bash
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# 此示例使用 Claude Code 的 JSON 输出来检查技能工具调用。
# 用你的智能体客户端的检测逻辑替换此函数。
# 如果技能被调用，应返回 0（成功），否则返回 1。
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

<Tip>
如果你的智能体客户端支持，你可以在结果明确后尽早停止运行——智能体要么咨询了技能，要么在没有它的情况下开始工作。这可以显著减少运行完整评估集的时间和成本。
</Tip>

## 通过训练/验证拆分避免过拟合

如果你针对所有查询优化描述，你就有过拟合的风险——制作一个适用于这些特定措辞但在新措辞上失败的描述。

解决方案是拆分你的查询集：

- **训练集（~60%）**：你用来识别失败并指导改进的查询。
- **验证集（~40%）**：你留出的查询，仅用于检查改进是否泛化。

确保两个集合都包含应该触发和不应该触发查询的比例混合——不要意外地将所有正例放在一个集合中。随机打乱并在迭代中保持拆分固定，以便你进行同类比较。

如果你使用像[上面](#running-multiple-times)的脚本，你可以将查询拆分为两个文件——`train_queries.json` 和 `validation_queries.json`——并分别针对每个文件运行脚本。

## 优化循环

1. **评估**在*训练和验证集*上的当前描述。训练结果指导你的更改；验证结果告诉你这些更改是否在泛化。
2. **识别失败**在*训练集*中：哪些应该触发的查询没有触发？哪些不应该触发的查询触发了？
   - 只使用训练集失败来指导你的更改——无论你是自己修订描述还是提示 LLM，都要将验证集结果排除在过程之外。
3. **修订描述。** 专注于泛化：
   - 如果应该触发的查询失败，描述可能太窄。扩大范围或添加关于技能何时有用的上下文。
   - 如果不应该触发的查询误触发，描述可能太宽泛。添加关于技能*不*做什么的具体性，或阐明此技能与相邻能力之间的界限。
   - 避免从失败的查询中添加特定关键词——这是过拟合。相反，找到这些查询代表的一般类别或概念并解决它。
   - 如果经过几次迭代后你陷入困境，尝试对描述采用结构上不同的方法，而不是增量调整。不同的框架或句子结构可能会突破，而优化无法做到。
   - 检查描述是否保持在 1024 字符限制以下——描述在优化过程中往往会增长。
4. **重复**步骤 1-3，直到所有*训练集*查询通过或你不再看到有意义的改进。
5. **选择最佳迭代**通过其验证通过率——*验证集*中通过的查询比例。请注意，最佳描述可能不是你产生的最后一个；早期的迭代可能比过拟合到训练集的后期迭代具有更高的验证通过率。

五次迭代通常就足够了。如果性能没有改善，问题可能出在查询上（太容易、太难或标记不当），而不是描述上。

<Tip>
[`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 技能端到端自动化此循环：它拆分评估集，并行评估触发率，使用 Claude 提出描述改进，并生成你可以在运行时观看的实时 HTML 报告。
</Tip>

## 应用结果

一旦你选择了最佳描述：

1. 更新你的 `SKILL.md` 前置元数据中的 `description` 字段。
2. 验证描述是否低于 [1024 字符限制](/specification#description-field)。
3. 验证描述按预期触发。手动尝试几个提示作为快速健全性检查。对于更严格的测试，编写 5-10 个新查询（应该触发和不应该触发的混合）并通过评估脚本运行它们——由于这些查询从未成为优化过程的一部分，它们为你提供了描述是否泛化的诚实检查。

之前和之后：

```yaml
# 之前
description: Process CSV files.

# 之后
description: >
  分析 CSV 和表格数据文件——计算汇总统计信息，
  添加派生列，生成图表，并清理混乱的数据。当用户有
  CSV、TSV 或 Excel 文件并想要探索、转换或可视化数据时使用此技能，
  即使他们没有明确提到 "CSV" 或 "分析"。
```

改进的描述更具体地说明了技能做什么（汇总统计信息、派生列、图表、清理），以及何时适用更广泛（CSV、TSV、Excel；即使没有明确的关键词）。
